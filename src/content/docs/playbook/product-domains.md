# Product Domains

The product domain map shows how the product is organised into ownership areas, which team owns each, and how the domains relate.

This page is intentionally a **pointer**, not the map itself. Domain maps are best maintained in a tool where boxes can be moved, connections drawn, and ownership annotations stay current — whichever diagramming or docs tool your team uses (Figma, Miro, Notion, Lucidchart, etc.). Pasting it as static markdown leads to a stale map within weeks.

## What a good product domain map looks like

- **One box per ownable domain.** A domain is something a single team can be accountable for. If two teams each own half, it's two domains.
- **Each box names the team.** No anonymous boxes.
- **Lines show dependency or data flow**, not org-chart relationships.
- **One column for the customer-facing surface, one for the underlying platform.** If you can't tell which is which, the map is too abstract.
- **Updated quarterly.** Even if nothing changed, mark it reviewed so the team knows.

## Where to put it

Maintain the live map in your team's primary working tool. Link to it from here:

> Domain map: `[link to your team's domain map]`

When the structure changes, update the link and the date. The map is a teaching tool — keep it current.
